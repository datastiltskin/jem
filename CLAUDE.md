# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

**Judiciary Entity Map (India) — JEM** is an open structural map of India's ~1,500 judicial and quasi-judicial institutions. It maps appointment chains, funding flows, complaint pathways, oversight mechanisms, and structural gaps — not case outcomes.

The pipeline is: YAML entity files → validate → derive scores → build `graph.json` → vanilla JS frontend.

## Commands

All pipeline commands run from `jem/` (not repo root):

```bash
cd jem
pip install -r scripts/requirements.txt   # pydantic>=2.0.0, pyyaml>=6.0

python scripts/validate.py                # Validate all YAML (exit 0 = clean)
python scripts/validate.py --strict       # Also fail on missing source URLs
python scripts/validate.py --entity data/entities/path/to/file.yaml  # Single file
python scripts/validate_graph_refs.py     # Check relationship ID references
python scripts/derive.py                  # Compute scores → data/derived/scores.yaml
python scripts/derive.py --explain <id>   # Score breakdown for one entity
python scripts/derive.py --clog-report    # Show clog severity summary
python scripts/derive.py --gaps-only      # Show structural gap summary
python scripts/build.py                   # Build repo-root graph.json (⚠ overwrites)
./scripts/build_safe.sh                   # Build to build/graph.staging.json only (safe)
./scripts/safe_pipeline.sh                # validate + derive + build in one shot
```

**Standard data session order:** validate → edit YAML → validate again → validate_graph_refs → derive → build → commit with `git add jem/data graph.json`.

**Local preview:** `cd jem/web && python3 -m http.server 8080` (requires `web/public/graph.json` symlink to resolve to repo-root `graph.json`).

## Architecture

### Data pipeline

```
jem/data/entities/**/*.yaml    One YAML file per entity
jem/data/relationships/        Typed relationship files (appellate, funding, appointment, …)
jem/data/derived/scores.yaml   Auto-generated by derive.py — do not edit manually
        ↓
scripts/validate.py            Pydantic v2 schema validation
scripts/derive.py              Computes independence_risk_score, discretionary_power_score,
                               structural_health_score from structural data fields
scripts/build.py               Merges entities + scores + relationships → graph.json
        ↓
graph.json (repo root)         Single JSON artifact consumed by the frontend
jem/web/public/graph.json      Symlink to repo-root graph.json
```

### Frontend

`jem/web/` — no build step, no bundler. Vanilla HTML + JS + CSS. CDN deps (D3, etc.) from jsdelivr/cdnjs. Key modules:

- `renderer.js` — D3 graph layout and draw loop
- `nodeShapes.js` / `nodeVisuals.js` — per-entity shape and styling rules
- `state.js` — global app state (zoom level, filters, selected entity)
- `panel.js` — right-side entity detail card
- `timeline.js` — time scroller (1950 → present, filters by created/abolished year)
- `main.js` — entry point, wires everything together

### Semantic zoom levels

| Level | What renders | Trigger |
|-------|-------------|---------|
| L0 Constellation | 8 cluster blocks, aggregate metrics | Default / fully zoomed out |
| L1 Backbone | Individual entity nodes, DAG layout | Scroll in |
| L2 Sub-entity | Sub-entities, edge labels | Scroll in further |
| L3 Detail | Full entity card, sources, score breakdown | Click entity |

### Entity YAML schema

Canonical schema: `entity_schema.yaml` (repo root). Key fields:

- `id`, `name`, `type`, `cluster`, `level_of_government`, `created_year`, `operational_status`, `data_quality`, `sources` — **required**
- `derived.*` — auto-computed by `derive.py`; never set manually
- `data_quality` values: `verified` | `complete` | `partial` | `unverified` | `contested` — controls visual weight in the renderer
- `operational_status`: `Active` | `Not_Constituted` | `Partial_Operational` | `De_Facto_Blocked` | `Abolished` | `Merged` | `Suspended`

### Score computation

`derive.py` computes three scores from structural fields:
- **independence_risk_score** — structural conflicts of interest (appointer = litigant, etc.)
- **discretionary_power_score** — opacity of criteria, reappointment risk, removal difficulty
- **structural_health_score** (0.0–1.0) — composite: ≤0.3 = critical, 0.3–0.6 = amber, 0.6–0.8 = yellow, ≥0.8 = green

Governance officeholder nodes (`president_india`, `chief_justice_india`, etc.) are excluded from scoring.

### Generators (high-risk, use sparingly)

| Script | Risk |
|--------|------|
| `generate_v1_states_bundle.py` | Can overwrite hand-edited entity YAML in `data/entities/_generated/` |
| `bootstrap_tn_district_lattice.py` | Idempotent — skips existing files |
| `merge_njdg_snapshot.py --apply` | Overwrites `case_volume` fields on matched entities |

## CI

GitHub Actions (`.github/workflows/validate.yml`) runs on every PR touching `jem/data/**` or `jem/scripts/**`: `validate.py --strict` → `derive.py` → `validate_graph_refs.py`. Does **not** deploy.

## Overwrite risks

- `build.py` (no `--output`) writes repo-root `graph.json` directly. Always verify `meta.entity_count` after: `python3 -c "import json; print(json.load(open('graph.json'))['meta'])"`
- Use `build_safe.sh` for experiments; diff `build/graph.staging.json` vs root before promoting.
- Never commit `_deploy_bundle/` or maintainer-only scripts (`build_friedso_deploy_bundle.sh`, `publish_github.sh` — gitignored).

## Data rules

- Every field that affects a score or gap marker requires a primary source (Constitution, statute, judgment, GoI report).
- `data_quality: verified` only when a direct GoI primary source URL is included.
- No case outcomes. No individual judge names. No editorial commentary.
- Relationships (appointment, appellate, funding arcs) are maintainer-only; contributors may submit data-quality upgrades on existing entities.
